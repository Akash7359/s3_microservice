Project Overview

This project implements a clean, layered FastAPI backend for handling file uploads. Uploaded files are persisted as objects in AWS S3, while structured metadata (filename, size, owner, path, soft-delete flag, timestamps) is stored in a MySQL database via the SQLAlchemy ORM.

The codebase is organised into four principal layers — API, Service, Repository, and Data — each with a single, well-defined responsibility. Alembic handles schema migrations and python-dotenv loads runtime configuration from .env.

Architecture Layers

Entry Layer
app/main.py
  • Creates the FastAPI application instance
  • Registers the upload router
  • Sets global middleware / CORS if required
API Layer
app/api/v1/s3_routes.py
  • Exposes POST /api/v1/files/upload
  • Accepts user_id, module_name, file
  • Injects DB session via Depends(get_db)
  • Delegates to FileService
Service Layer
app/services/file_service.py
  • Validates file size vs MAX_FILE_SIZE
  • Builds S3 key: S3_service/{year}/{month}/{user_id}/{module}/{filename}
  • Uploads file stream via S3Repository
  • Persists metadata via FileRepository
Repository Layer
app/repositories/
  • S3Repository — upload, delete, presigned URL
  • FileRepository — insert, fetch by id, soft delete
  • Abstracts all external I/O from business logic
Data Layer
app/models/ · app/schemas/
  • ORM model maps to files table
  • Pydantic schema serialises API response
  • Alembic manages migrations
Config & Infra
app/core/ · alembic/
  • config.py loads .env vars
  • database.py creates engine & SessionLocal
  • Alembic manages schema migrations

End-to-End Runtime Flow

  1. Client sends POST /api/v1/files/upload with a multipart body including user_id, module_name, and the binary file.

  2. FastAPI router parses the request and injects a database session via Depends(get_db).

  3. Router instantiates FileService(db) and calls upload_file(user_id, module_name, file).

  4. Service seeks to EOF to measure the stream size, raises HTTP 413 if size > MAX_FILE_SIZE, then resets the cursor to position 0.

  5. Service extracts the file extension and constructs the S3 object key: S3_service/{year}/{month}/{user_id}/{module_name}/{filename}.

  6. Service calls S3Repository.upload_file(stream, bucket, key) which invokes boto3.upload_fileobj.

  7. Service builds a File ORM entity populated with all metadata fields (user_id, module_name, filename, extension, size, s3_path, bucket_name).

  8. FileRepository.create(file_obj) runs db.add → db.commit → db.refresh and returns the persisted row.

  9. The persisted ORM object is serialised via the FileResponse Pydantic schema.

  10. Router returns an HTTP 200 JSON response containing the file metadata to the client.

idINT PK AINoAuto-incrementing primary key

module_nameVARCHARNoSource module / context (e.g. invoices)

S3 Key Format

Object Key Pattern
S3_service/{year}/{month}/{user_id}/{module_name}/{original_filename}

Example: S3_service/2025/06/42/invoices/receipt_q2.pdf

API Response Schema — FileResponse

Returns: id, user_id, module_name, filenames, extension, size, s3_path, bucket_name, created_at. Excludes internal flags like is_deleted.

Gaps & Risks Identified

  • HIGH
    Missing app/dependencies.py module

    s3_routes.py imports app.dependencies.get_db, but the file does not exist. The application will fail to start with an ImportError on any environment.

  • HIGH
    No HTTP exception handling

    Unhandled S3 errors (e.g. NoCredentialsError, BucketNotFound) or DB errors will surface as unformatted 500 responses. Proper HTTPException wrappers are missing throughout the service and repository layers.

  • MED
    Alembic target_metadata = None

    alembic/env.py has no reference to Base.metadata. Auto-generated migrations will be empty, meaning schema drift will go undetected.

  • MED
    Alembic DB URL is a placeholder

    alembic.ini still contains driver://user:pass@localhost/dbname. Migrations will not run until this is replaced or dynamically read from .env.

  • MED
    No initial migration version

    The alembic/versions/ folder has no migration files. The files table must be tracked via an initial revision before deploying to any environment.

  • LOW
    No API tests

    There are no unit or integration tests for upload success or failure paths. Test coverage for the service and repository layers is also absent.

  • LOW
    S3 delete not called on DB rollback

    If the file is successfully uploaded to S3 but the subsequent DB commit fails, the orphaned S3 object is never cleaned up. A compensating delete call should be added in the exception handler.

Next Implementation Steps

Step 1 Create app/dependencies.py

Add a get_db() generator that yields a SessionLocal instance and closes it after the request.

Step 2 Wire Alembic to Base.metadata

Import Base from app.core.database in alembic/env.py and set target_metadata = Base.metadata.

Step 3 Configure Alembic DB URL from env

Replace the placeholder in alembic.ini and override config.set_main_option in env.py using settings.DATABASE_URL.

Step 4 Generate initial migration

Run alembic revision --autogenerate -m "create_files_table" and review the generated script before applying.

Step 5 Add structured error handling

Wrap S3 and DB operations in try/except blocks. Raise typed HTTPExceptions (413, 500) and add an S3 compensating delete on DB failure.

Step 6 Write API tests

Use pytest + TestClient with a mocked S3 client (moto) and an in-memory SQLite DB to cover upload success, file-too-large, and S3 failure paths.
